# HHSDD 系统架构与设计哲学

**Hyper-Hierarchical Spawn-Driven Decision** / 超分层派生驱动决策系统

## 一、整体设计哲学：资源约束下的自主进化

系统围绕一个核心命题构建：**如何让 LLM Agent 在有限的上下文窗口（token 预算）内，自主完成复杂任务，并通过执行反馈持续优化自身行为**。设计哲学可概括为三点：

1. **能量即 Token**：将所有资源（上下文长度、API 调用成本、计算时间）统一量化为“能量”，与 token 直接挂钩，使 Agent 的每一步决策都具有真实的经济约束。
2. **缓存优先的上下文管理**：通过栈式分层和消息前缀复用，最大化 API 缓存命中率，将长期任务的边际成本降至最低。
3. **群体智能驱动的元优化**：蚁群、粒子群、萤火虫、狼群等算法不参与任务执行，而是在元层面优化参数、合并计划、分工角色，以极低开销换取系统长期效率提升。

## 三、能量系统：Token 本位的双层预算

能量系统是整个架构的**硬约束引擎**，设计要点：

### 1. 能量定义
- `总能量 = CONTEXT_BUDGET - CONTEXT_RESERVE`（约 154,800 token）
- 所有操作消耗均换算为 token：
  - LLM 调用：`input_tokens × 1.0 + output_tokens × 3.0`（输出成本更高）
  - 缓存命中：`cache_read_tokens × 0.1`，写入 `× 1.25`
  - 命令执行：预扣 `预估耗时 × 退款速率`，实际完成后退还差额
  - 每步固定开销：`STEP_OVERHEAD`（1000 token）

### 2. 双层约束
| 约束类型 | 变量 | 行为 |
|----------|------|------|
| **软约束（流动资金）** | `energy` | 可预扣、返还、奖励，允许临时透支（下限 -10%） |
| **硬约束（总消耗）** | `total_spent` | 不可逆累加，触顶即停 |

### 3. 投资-收益模型
- **探索模式**：成功返还本金 + ROI 奖金（如 +50%），失败血本无归。
- **利用模式**：成功返还本金，失败返还 50%。
- **Skill 押金**：使用 Skill 预扣押金，任务成功全额返还，失败没收。

### 4. Spawn 能量隔离
父 Agent 派生子 Agent 时，扣留当前能量的 **15%** 作为保底，子 Agent 在剩余 85% 内独立运行，返回后释放保底能量。这防止了子树耗尽全局预算。

## 四、上下文管理：栈式分层 + 缓存友好消息构建

### 1. 栈帧模型
```
[0] constraint (level=0)  ← 系统提示词 + Skills，永久保留
[1] plan (level=1)        ← 任务计划（含反向推理），极少修改
[2] step_detail/summary (level=2) ← 执行轨迹，动态压缩
[3] merge (level=3)       ← 阶段汇合总结
```
- **固定底**：`constraint` 和 `plan` 作为缓存前缀，几乎不变。
- **可压缩顶**：每步后将详细日志压缩为 100 字摘要，超过阈值时触发弹栈回收。

### 2. 缓存优化
- 子 Agent 继承父 Agent 的消息前缀（`system_blocks` + 计划 + 已完成步骤）。
- 工具定义通过 `cache_control: ephemeral` 标记缓存断点。
- 最新 user 消息单独标记断点，形成三级缓存结构：
  ```
  [System Blocks] → [Plan] → [Completed Steps] → [Latest User]
      断点1            断点2         (动态)           断点3
  ```

### 3. 自适应回收
当能量不足以支付预估的下次 API 成本时，按优先级弹出栈帧：
1. 空壳结果（≤50 字符）
2. 已压缩的 `summary`/`merge` 帧
3. 详细 `step_detail` 帧
回收的 token 补充回流动资金。

## 五、群体智能算法的角色

| 算法 | 集成位置 | 作用 | 开销 |
|------|----------|------|------|
| **蚁群（信息素）** | `ExperienceStore` | 成功任务路径的信息素增强，失败蒸发；检索时按 `pheromone × rank` 排序，推荐高频成功路径 | 极低（SQL 乘法） |
| **粒子群（PSO）** | `pso_tuner.py` | 每 10 个任务迭代一次，优化 `cost_step`、`cost_tool`、`explore_roi` 等参数，最大化 `成功率×10 + 效率×5` | 极低（纯数值计算） |
| **萤火虫（子任务合并）** | `Agent._merge_subtasks` | 计算子任务描述相似度，高亮度（成功率高+特异性强）吸引低亮度，合并冗余步骤 | 极低（Jaccard 相似度） |
| **狼群（动态分工）** | `BayesianEnergyManager.get_role_probability` | 根据能量比例和任务进展动态计算 `explore` 概率：能量充足+进展缓慢→多探索；能量紧张+接近完成→多利用 | 极低（线性公式） |

## 六、关键设计决策与工程权衡

### 1. 为什么能量必须锚定 Token？
早期版本按“步数”固定扣能，导致简单任务与复杂任务成本相同，奖励信号失真。改为 token 挂钩后，Agent 自发产生“少说废话、精简输出”的行为，能量消耗与真实 API 费用对齐。

### 2. 为什么保留“紧急交付”？
即使能量耗尽，若已执行了部分子任务，系统会使用 `bypass_energy` 调用一次轻量 LLM，将已有产物拼接为部分报告，确保用户至少得到一个可读的交付物。这是用户体验的兜底，而非能量约束的破坏。

### 3. 为什么子 Agent 继承父消息前缀？
若不继承，每次派生都会重新发送完整上下文，缓存命中率为零，成本指数爆炸。继承前缀后，分支探索的增量成本极低，使大规模递归分解成为可能。

### 4. 为什么用栈而非向量库管理短期记忆？
栈的层级结构天然匹配任务的分解与汇合，且与 API 消息列表的顺序一致。向量库适合长期跨任务检索，栈适合当前任务内的时序回溯，二者互补。

## 七、数据流与状态持久化

- **`state.json`**：保存栈帧、步骤计数器、子任务队列，支持中断恢复。
- **`pending_exp.json`**：待处理经验元数据，累积到阈值后批量调用 LLM 提取教训并入库。
- **`experience.db`**（SQLite + FTS5）：存储历史任务摘要、教训、信息素，支持全文检索。
- **`flowchart.md`**：Mermaid 流程图，实时记录任务执行拓扑，便于调试和可视化。

## 八、设计哲学总结

> **“用工程约束倒逼智能涌现”**

系统并未试图让 LLM 变得更聪明，而是通过精心设计的**能量管道**、**缓存阶梯**和**群体反馈**，迫使 Agent 在受限资源下做出高效决策。每一次 API 调用都有明确的 token 成本，每一次派生都有风险与回报的权衡，每一次成功都会通过信息素和 PSO 强化未来行为。最终，Agent 的行为不再是黑盒，而是可解释、可审计、可持续优化的工程系统。
