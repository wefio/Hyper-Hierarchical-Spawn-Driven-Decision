# HHSDD 系统架构与设计哲学

**Hyper-Hierarchical Spawn-Driven Decision** / 超分层派生驱动决策系统

## 一、整体设计哲学：资源约束下的迭代优化

系统围绕一个核心命题构建：**如何让 LLM Agent 在有限的上下文窗口（token 预算）内，自主完成复杂任务，并通过执行反馈持续优化自身行为**。

1. **能量即 Token**：将所有资源（上下文长度、API 调用成本、计算时间）统一量化为“能量”，与 token 直接挂钩，使 Agent 的每一步决策都具有真实的经济约束。同时可以阻止agent无限递归但不干活。
2. **缓存优先的上下文管理**：通过栈式分层和消息前缀复用，最大化 API 缓存命中率，将长期任务的边际成本降至最低。

## 三、能量系统：Token 本位的双层预算

能量系统是整个架构的**硬约束引擎**，设计要点：

### 1. 能量定义
- `总能量 = CONTEXT_BUDGET - CONTEXT_RESERVE`（约 154,800 token）
- 所有操作消耗均换算为 token：
  - LLM 调用：`input_tokens × 1.0 + output_tokens × 3.0`（输出成本更高）
  - 缓存命中：`cache_read_tokens × 0.1`，写入 `× 1.25`
  - 命令执行：预扣 `预估耗时 × 退款速率`，实际完成后退还差额
  - 每步固定开销：`STEP_OVERHEAD`（1000 token）

### 2. 双层约束
| 约束类型 | 变量 | 行为 |
|----------|------|------|
| **软约束（流动资金）** | `energy` | 可预扣、返还、奖励，允许临时透支（下限 -10%） |
| **硬约束（总消耗）** | `total_spent` | 不可逆累加，触顶即停 |

### 3. 投资-收益模型
- **探索模式**：成功返还本金 + ROI 奖金（如 +50%），失败血本无归。
- **利用模式**：成功返还本金，失败返还 50%。
- **Skill 押金**：使用 Skill 预扣押金，任务成功全额返还，失败没收。

### 4. Spawn 能量隔离
父 Agent 派生子 Agent 时，扣留当前能量的 **15%** 作为保底，子 Agent 在剩余 85% 内独立运行，返回后释放保底能量。这防止了子树耗尽全局预算。

## 四、上下文管理：栈式分层 + 缓存友好消息构建

### 1. 栈帧模型
```
[0] constraint (level=0)  ← 系统提示词 + Skills，永久保留
[1] plan (level=1)        ← 任务计划（含反向推理），极少修改
[2] step_detail/summary (level=2) ← 执行轨迹，动态压缩
[3] merge (level=3)       ← 阶段汇合总结
```
- **固定底**：`constraint` 和 `plan` 作为缓存前缀，几乎不变。
- **可压缩顶**：每步后将详细日志压缩为 100 字摘要，超过阈值时触发弹栈回收。

### 2. 缓存优化
- 子 Agent 继承父 Agent 的消息前缀（`system_blocks` + 计划 + 已完成步骤）。
- 工具定义通过 `cache_control: ephemeral` 标记缓存断点。
- 最新 user 消息单独标记断点，形成三级缓存结构：
  ```
  [System Blocks] → [Plan] → [Completed Steps] → [Latest User]
      断点1            断点2         (动态)           断点3
  ```

### 3. 自适应回收
当能量不足以支付预估的下次 API 成本时，按优先级弹出栈帧：
1. 空壳结果（≤50 字符）
2. 已压缩的 `summary`/`merge` 帧
3. 详细 `step_detail` 帧
回收的 token 补充回流动资金。

## 五、关键设计决策与工程权衡

### 1. 为什么能量锚定 Token？
目标是在有限的上下文中解决复杂任务，所以自然需要一个上下文预算。早期版本按Token映射固定点数的能量，后来觉得太麻烦了，不如直接1:1。

### 2. “紧急交付”的设计原因？
因为早期实验中Agent完成了任务但没有交付成果，因为能量用完了。加入紧急交付后即使能量耗尽，若已执行了部分子任务，系统会使用 `bypass_energy` 调用一次轻量 LLM，将已有产物拼接为部分报告，确保用户至少得到一个可读的交付物。这是用户体验的兜底，而非能量约束的破坏。

### 3. 为什么子 Agent 继承父消息前缀？
提高缓存命中。若不继承，每次派生都会重新发送完整上下文，缓存命中率为零，成本指数爆炸。继承前缀后，分支探索的增量成本极低，使大规模递归分解成为可能。网页版的大模型有个重新生成回答的功能，思想应该差不多。

### 4. 为什么用栈？
栈的层级结构天然匹配缓存命中、任务的分解与汇合。本项目的思考是从栈结构开始的。

## 六、插件系统

Agent 通过 `_PLUGINS` 列表动态加载外部模块，将领域特定逻辑（工具、验证、流程兜底）从通用框架中解耦。

```
agent.py
  ├── _load_plugins()          # importlib 动态加载
  ├── _PLUGINS: list[module]   # 已加载的插件列表
  │
  ├── Agent.__init__
  │     └── plugin.on_agent_init(self, bus)
  ├── Agent.run()
  │     └── plugin.on_run_start(self, task)
  ├── ToolExecutor.execute()
  │     ├── plugin.get_tool_definitions()   → 合并工具
  │     ├── plugin.get_tool_handlers()      → 合并 handler
  │     └── plugin.on_tool_executed(...)    → 后处理
  └── 子任务循环
        └── plugin.on_subtask_loop(...)     → 反馈注入
```

**加载流程**：`Config.PLUGINS`（逗号分隔）→ `importlib.import_module()` → 检查接口函数 → 注册到对应位置。

**任务验证扩展**：插件通过 `task_verifier.register_category()` 和 `task_verifier.register_verifier()` 注册领域验证器，验证框架通过 `__getattr__` 动态分派到注册的函数。

## 七、数据流与状态持久化

- **`state.json`**：保存栈帧、步骤计数器、子任务队列，支持中断恢复。
- **`pending_exp.json`**：待处理经验元数据，累积到阈值后批量调用 LLM 提取教训并入库。
- **`experience.db`**（SQLite + FTS5）：存储历史任务摘要、教训、权重，支持全文检索。
- **`flowchart.md`**：Mermaid 流程图，实时记录任务执行拓扑，便于调试和可视化。
- **`archive/`**（Pointer 磁盘缓存）：将即将被淘汰的上下文持久化到 `.md` 文件，维护 `index.json` 索引，支持按 ID/关键词/作用域召回。
